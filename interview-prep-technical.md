# 인터뷰 예상 질문 & 답변 - 기술 역량

## Q1. 대용량 트래픽을 어떻게 처리하셨나요?

**답변:**
누적 회원 1700만, MAU 400만 이상의 대규모 서비스를 운영하면서 다양한 대용량 트래픽 처리 경험을 쌓았습니다.

**가상 순번 대기 시스템 사례:**
- Redis Sorted-Set 기반 대기열·순번 관리로 동시접속 100K 이상 안정 제어
- Lua 스크립트로 단일 키 트랜잭션·락 최소화 설계
- Sliding-Window로 트래픽 대기/진입 처리 제어
- HTTP Long-Term Polling과 서버 계산 기반 Polling 주기 동적 조절
- 단일 인스턴스 기준 TPS 600, p95 응답시간 3ms 달성
- AWS 클라우드 인프라 활용한 Auto-Scaling 환경 구성

**민생회복 소비쿠폰 시스템:**
- 최대 TPS 1400 대응 가능한 시스템 구축
- 개인정보 암호화 처리 및 불필요한 DB Lock 제거로 성능 최적화

## Q2. 성능 개선 경험을 구체적으로 설명해주세요.

**답변:**
크게 두 가지 대표적인 성능 개선 사례가 있습니다.

**1. 가맹점 연동 Batch 성능 최적화:**
- 처리 시간: 4300s → 24s (약 179배 개선)
- 병목 구간 식별: 파일 IO/DB/외부 API 호출 분석
- Spring Batch Partitioner로 파티셔닝·Step 병렬 처리
- 외부 API 연동을 WebClient 전환하여 Async/Non-Blocking 처리
- Chunk Listener로 후속 비동기 호출 구현
- 재시도 정책 정교화로 운영 안정성 확보

**2. 주소 검색 시스템 최적화:**
- 도로명 검색 성공율: 89.2% → 99.8%
- 지번/관련-지번 검색 성공율: 73.3% → 81.3%
- 평균 응답시간: 18ms → 6ms (3배 개선)
- Filebeat → Logstash → Elasticsearch 데이터 파이프라인 구성
- 도로명/지번/동의어 기반 맞춤 애널라이저 설계
- 유사도 매칭 로직 적용 및 좌표 정합성 검증

## Q3. MSA 환경에서 개발한 경험이 있나요?

**답변:**
코나아이의 결제플랫폼은 MSA 구조로 구성되어 있으며, 다양한 금융 관련 도메인으로 이루어져 있습니다.

**주요 개발 경험:**
- **OpenAPI Gateway 시스템:** Spring Cloud Gateway, Spring Cloud Circuit Breaker 활용하여 API 인증/인가, 데이터 암복호화, 장애 전파 방지 처리
- **Reactive 프로그래밍:** Spring WebFlux 활용한 비동기 Non-Blocking 처리
- **가상계좌 관리 서비스:** 은행사 TCP 전문 연동 프로세스 구축
- **외부 제휴사 연동:** 실시간 거래 내역 연동 API 및 Batch 프로그램 개발

각 도메인 간 느슨한 결합을 유지하면서도 비즈니스 요구사항을 충족시키기 위해 메시지 큐(RabbitMQ), 캐시(Redis) 등을 활용하여 안정적인 시스템을 구축했습니다.

## Q4. 데이터베이스 최적화 경험이 있나요?

**답변:**
대용량 데이터를 다루면서 다양한 DB 최적화 경험을 쌓았습니다.

**주요 경험:**
- **대규모 데이터 이관:** 부산 동백전 300만 회원, 3억 거래 내역 / 울산페이 53만 회원, 1억 거래 내역 등 대규모 데이터 마이그레이션
- **불필요한 DB Lock 제거:** 민생회복 소비쿠폰 시스템 개선 시 조회 로직에서 불필요한 Lock 제거로 성능 향상
- **인덱스 설계:** 회원 정보 조회 성능 최적화를 위한 인덱스 설계
- **Elasticsearch 활용:** 주소 검색 시스템에서 도로명/지번/동의어 기반 맞춤 애널라이저 설계로 검색 성공률 향상
- **캐싱 전략:** NONCE 발급 데이터 Redis 캐싱으로 조회 성능 최적화

## Q5. 보안을 고려한 개발 경험이 있나요?

**답변:**
금융 및 결제 플랫폼 분야에서 10년간 일하면서 보안을 항상 최우선으로 고려해왔습니다.

**주요 보안 강화 사례:**
- **회원 인증 프로세스 보안 강화:** NONCE 발급 및 검증 체계 수립으로 본인 인증 검증 데이터 재사용 방지, 로그인 비밀번호 암호화 데이터 재사용 방지
- **개인정보 보호:** 민생회복 소비쿠폰 시스템에서 개인정보 암호화 처리
- **ISMS-P 준수:** 탈퇴 회원 정보 분리 보관 및 보존 기간 만료 회원 정보 파기 프로세스 구축
- **KYC 체계 수립:** 회원 식별을 위한 KYC 체계 수립 및 AML 미수행/위반 회원 서비스 이용 차단
- **API 보안:** OpenAPI Gateway에서 Access Key 인증, HTTP Body 암복호화, Call-Count 제한 처리

코드 작성 시에도 OWASP Top 10 취약점(SQL Injection, XSS, Command Injection 등)을 항상 고려하며, 발견 시 즉시 수정합니다.

## Q8. 프로젝트에서 가장 어려웠던 기술적 챌린지는 무엇이었나요?

**답변:**
가상 순번 대기 시스템 개발이 가장 도전적인 프로젝트였습니다.

**챌린지:**
- 동시접속 100K 이상의 대규모 트래픽을 안정적으로 제어
- 실시간 대기열 순번 관리 및 공정한 진입 처리
- 좀비 토큰 자동 만료 등 운영 자동화
- p95 응답시간 3ms 이하의 초저지연 요구사항

**해결 과정:**
1. **아키텍처 설계:** Redis Sorted-Set 기반 대기열 관리 구조 설계
2. **성능 최적화:** Lua 스크립트로 단일 키 트랜잭션 처리, 락 최소화
3. **트래픽 제어:** Sliding-Window 알고리즘으로 진입 제어
4. **사용자 경험 개선:** HTTP Long-Term Polling + 서버 계산 기반 동적 Polling 주기 조절
5. **모니터링:** 실시간 트래픽 대기 현황 모니터링 및 통계 데이터 관리
6. **확장성:** AWS Auto-Scaling 환경 구성

**결과:**
- 분당 46K 요청 처리
- 단일 인스턴스 TPS 600
- p95 응답시간 3ms
- 월 1일 대규모 이벤트 기간 코어 시스템 지연 장애 0건

이 프로젝트를 통해 대용량 분산 시스템 설계, Redis 활용, 성능 최적화 등 다양한 기술적 역량을 크게 향상시킬 수 있었습니다.

## Q10. 아키텍처 설계 경험이 있나요?

**답변:**
대규모 시스템 설계 및 아키텍처 개선 경험이 있습니다.

**설계 원칙 적용:**
- **헥사고날 아키텍처:** 도메인 로직과 인프라 관심사 분리
- **클린 아키텍처:** 의존성 방향 제어, 테스트 가능한 구조
- **DDD (Domain Driven Design):** 도메인 중심 설계, 유비쿼터스 언어 활용

**주요 설계 경험:**
- **가상 순번 대기 시스템:** Redis 기반 분산 시스템 아키텍처 설계 (요구정의 → 설계 → 개발 → 테스트 → 운영 전 과정 주도)
- **OpenAPI Gateway:** Spring Cloud Gateway 기반 API Gateway 아키텍처 설계
- **주소 검색 시스템:** Elasticsearch 기반 검색 시스템 아키텍처 및 데이터 파이프라인 설계

**설계 시 고려사항:**
- 확장성: 향후 트래픽 증가에 대비한 확장 가능 구조
- 유지보수성: 코드 가독성, 모듈화, 의존성 최소화
- 성능: 응답시간, TPS, 자원 효율성
- 안정성: 장애 전파 방지, 모니터링, 자동화
